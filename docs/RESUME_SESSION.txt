PROBLEMES RESOLUS ET MODIFICATIONS APPORTEES
---------------------------------------------

PROBLEME 1 : STP bloque les chemins alternatifs
  Fichier : /home/debian/Bureau/topologies/topo_projet.py
  Fix : Suppression de stp=True et des commandes ovs-vsctl stp_enable=true
  Résultat : Tous les liens actifs, Ryu voit la topologie complète

PROBLEME 2 : Broadcast storm sans STP
  Fichier : /home/debian/Bureau/DID/did_controller.py
  Fix : Remplacement de OFPP_FLOOD par return (DROP) pour destinations inconnues
  Résultat : Plus de boucles, trafic contrôlé uniquement via flow rules

PROBLEME 3 : ARP ne fonctionne plus sans flood
  Fichier : /home/debian/Bureau/topologies/topo_projet.py
  Fix : Injection ARP statique pour les 22 hotes au démarrage
  Code ajouté :
    for i in range(1, 23):
        h = net.get(f'h{i}')
        for j in range(1, 23):
            if i != j:
                mac = '00:00:00:00:00:{:02x}'.format(j)
                h.cmd(f'arp -s 10.0.0.{j} {mac}')

PROBLEME 4 : Authentification DID échoue pour switches périphériques
  Fichier : /home/debian/Bureau/topologies/topo_projet.py
  Cause : Lancement simultané (&) de 22 agents auth -> Ryu surchargé, paquets perdus
  Fix : Lancement séquentiel (sans &), chaque auth attend la fin avant la suivante
  Code :
    for i in range(1, 23):
        h = net.get(f'h{i}')
        h.cmd(f'python3 {agent_path} switch_{i} > /dev/null 2>&1')

PROBLEME 5 : Auth lancée hors Mininet (full_auth.py externe)
  Cause : Tous les paquets UDP entraient par s1, anti-spoofing bloquait les autres
  Fix : Auth intégrée dans topo_projet.py, lancée depuis chaque hote h{i} via net.get()

PROBLEME 6 : Monitoring FLUX absent après installation des flow rules
  Fichier : /home/debian/Bureau/DID/did_controller.py
  Cause : Après le premier paquet, flow rules installées -> plus de PacketIn -> plus de FLUX
  Fix : Ajout idle_timeout=10s sur les flow rules de routage
  Résultat : Rules expirent après 10s d'inactivité -> prochain paquet regenere FLUX


ETAT FINAL DU SYSTEME
----------------------

Architecture :
  - Controleur : Ryu (did_controller.py) + ryu.topology.switches
  - Topologie : 22 switches, 22 hotes, liens maillés (topo_projet.py)
  - Auth : DID + ECDSA + Blockchain (blockchain.py, agent_auth.py)
  - Routage : Dijkstra via NetworkX, recalcul automatique

Comportement Zero Trust :
  - Switch non-authentifié : tout trafic droppé sauf paquets auth UDP:9999
  - Switch authentifié : flow rules installées, trafic autorisé
  - Switch perdu : retiré du graphe, routes recalculées automatiquement

Test S1 down - Routes recalculées :
  AVANT (S1 actif)    : s14 -> s11 -> s1 -> s2 -> s3  (4 hops)
  APRES (S1 quarant.) : s14 -> s11 -> s17 -> s19 -> s20 -> s21 -> s2 -> s3  (7 hops)

Monitoring FLUX (dans terminal Ryu) :
  FLUX | did:sdn:switch_14:9e739cb4 | 10.0.0.14 --> 10.0.0.3
  FLUX | did:sdn:switch_11:17d297d3 | 10.0.0.14 --> 10.0.0.3
  ... (un log par switch sur le chemin, dans les deux sens)

COMMANDES UTILES
----------------

Lancer Ryu :
  cd ~/Bureau/DID
  source ../depin_env/bin/activate
  ryu-manager --observe-links did_controller.py ryu.topology.switches

Lancer la topologie :
  sudo python3 ~/Bureau/topologies/topo_projet.py

Mettre un switch en quarantaine :
  mininet> sh ovs-vsctl del-controller s1

Restaurer un switch :
  mininet> sh ovs-vsctl set-controller s1 tcp:127.0.0.1:6633
  (puis relancer l'auth depuis Mininet : h1 python3 ~/Bureau/DID/agent_auth.py switch_1)

Voir les flow rules d'un switch :
  mininet> sh ovs-ofctl -O OpenFlow13 dump-flows s14
